# CLAUDE.md — Gptgraber

ТЗ и карта проекта для ассистента. Читать перед любой правкой. Язык общения и комментариев в коде — **русский**.

> **В репозитории ДВЕ версии.** §1–§7 описывают исходную **Firefox-версию**
> (`server-csharp/VoiceBridge` + расширение). Новая автономная **версия на WebView2**
> (`poc-webview2/`, «GPT Grabber»: встроенный ChatGPT, без Firefox/расширения/сети) и её
> установщик — в **§8**. Грабли §6 (DOM ChatGPT, вставка, фокус, индикатор) переиспользованы и там.

## 1. Что это и зачем

Голосовой мост: **диктовка в табе ChatGPT (Firefox) → текст автоматически вставляется в активное окно** (любое — VS Code, редактор, мессенджер). Идея — использовать качественную диктовку ChatGPT как системный голосовой ввод куда угодно.

Управление — одной клавишей в стиле WhisperFlow. Пользователь сидит в своём приложении, не переключаясь в браузер; расширение само кликает кнопки диктовки в ChatGPT, читает результат и отдаёт серверу, сервер вставляет текст в активное окно.

## 2. Модель управления (ГЛАВНОЕ)

Хоткеи ловит **низкоуровневый хук клавиатуры в C#-сервере** (не `RegisterHotKey` — он не умеет «только модификаторы»). Жест распознаётся на **отпускании** комбо: пока зажаты Ctrl+Win, хук следит, нажималась ли клавиша «+буфер» и не нажималась ли посторонняя клавиша (тогда жест отменяется).

- **Ctrl+Win** — тоггл диктовки:
  - из `Idle`: сервер запоминает активное окно и переходит в `Preparing` — **сначала гарантирует готовность Firefox+таба ChatGPT** (см. §6.16): если расширение на связи — шлёт `ensureTab` (расширение откроет таб, если его нет) и ждёт `ready`; если Firefox не запущен — сам поднимает его. Получив `ready`, **на мгновение выносит окно Firefox на передний план** (иначе FF не стартует захват микрофона для фонового окна, см. §6.11), шлёт `mic` (клик «Start dictation»), состояние → `Recording`. Расширение, увидев, что запись пошла, шлёт `recording` → сервер **возвращает фокус** в рабочее окно. Если `ready` не пришёл за таймаут — старт отменяется (`Idle`).
  - из `Recording`: сервер запоминает **активное окно** (цель вставки), шлёт `stop` (клик «Submit dictation»), состояние → `Idle`. ChatGPT транскрибирует (1–3 с) → текст приходит → **авто-вставка в запомненное окно** (Ctrl+V) → `clear` (очистка композера).
- **Ctrl+Win+«Y»** — то же завершение, но текст **дополнительно остаётся в буфере обмена** (прежний буфер не восстанавливается). Клавиша «+буфер» ловится по **скан-коду `0x2C`** (физическая позиция, на немецкой раскладке «Y»), а НЕ по `vkCode` — vkCode зависит от раскладки (DE→0x59, RU→0x5A), скан-код одинаков. См. §6.12.
- Прочие `Ctrl+Win+<другое>` (D, стрелки) — игнорируются, не «глотаются» (системные шорткаты и Пуск работают).
- **Вне пары «старт/стоп» Ctrl+Win = ОТМЕНА** (во время подготовки, незапустившейся записи, распознавания после стопа) — всё сбрасывается в `Idle`, индикатор «Отменено». См. §6.20.

**Вставка не крадёт фокус по сути:** пользователь и так в целевом окне, вставка = Ctrl+V в уже активное окно. `Injector.ForceForeground` (вынос окна на передний план через `AttachThreadInput`) используется и для подъёма Firefox на старте, и как гарантия, что цель на переднем плане перед Ctrl+V.

**Сетевой режим (хаб) — расширение модели, не замена.** Сервер сам по себе остаётся полноценной конечной точкой: запущенный без аргументов `VoiceBridge` ловит локальный Ctrl+Win и вставляет на этой же машине, как раньше (это «локальный контроллер» внутри `ServerApp`, владелец `Id=0`). Поверх него сервер — это **хаб**: один Firefox-движок диктовки и сколько угодно **контроллеров**. Сетевой клиент (`VoiceBridge --connect <IP>` на другой машине, `ClientApp`+`WsClient`) ловит свой Ctrl+Win, по СТОПу захватывает **своё** активное окно и шлёт серверу `start`/`stop`; сервер гоняет тот же Firefox, а распознанный текст возвращает **владельцу сессии** — тому, кто её запустил (`inject` по WS → клиент вставляет у себя). Диктовка одна на один Firefox: пока идёт сессия одного владельца, старты других игнорируются («занято»). Трюк §6.11 (подъём окна FF, возврат фокуса) всегда выполняет сервер — окно FF на машине-сервере; для удалённой сессии захват окна-цели и сам Ctrl+V происходят на машине клиента. Защиты/токена нет — рассчитано на доверенную локальную сеть.

## 4. Сборка и запуск

**Сервер** (нужен .NET 10 SDK):
```powershell
cd server-csharp\VoiceBridge
dotnet build      # проверка
dotnet run        # запуск (или в Visual Studio: F5)
```
Окно консоли = сервер; держать открытым. Ctrl+C — выход. Порт `http://localhost:17890/` обычно открывается без админ-прав; если отказ — `netsh http add urlacl url=http://localhost:17890/ user=ДОМЕН\Пользователь`.

**Режимы запуска** (`--help` печатает справку). Запуск — через `dotnet run -- <аргументы>` или `dotnet bin\Debug\net10.0-windows\VoiceBridge.dll <аргументы>`. Свой `.exe` НЕ собираем (`UseAppHost=false`, см. §6.14 — SAC), поэтому всегда через `dotnet`:
```powershell
dotnet run                            # сервер на localhost:17890 (как раньше)
dotnet run -- --host +                # сервер слушает все интерфейсы (доступ по сети)
dotnet run -- --connect 192.168.1.50  # сетевой клиент к серверу-хабу (порт по умолч. 17890)
dotnet run -- --port 18000            # сменить порт (общий флаг для обоих режимов)
dotnet run -- --overlay-compact       # индикатор статуса без подписи (только значок)
dotnet run -- --no-overlay            # выключить индикатор статуса совсем
dotnet run -- --overlay-colors bg=1E1E20,rec=00C853   # перекрасить индикатор (ключи: bg border text wait rec busy err)
dotnet run -- --lang ru               # язык надписей (en|de|ru); по умолчанию — язык Windows
```
Для сетевого сервера разово (админ-консоль): `netsh http add urlacl url=http://+:17890/ user=ДОМЕН\Пользователь` и правило брандмауэра на входящий TCP 17890. На машине сетевого клиента Firefox/расширение НЕ нужны — только .NET-рантайм и эта DLL.

**Расширение** (разработка): `about:debugging#/runtime/this-firefox` → Load Temporary Add-on → `extension-firefox/manifest.json`. После правок — кнопка **Reload** там же; таб ChatGPT — **Ctrl+R**. Временное дополнение слетает при перезапуске браузера.

**Постоянная установка расширения** (нужна для ветки «запустить закрытый Firefox», §6.16). Полные шаги — `extension-firefox/README.md`. Кратко:
- **Dev Edition/ESR/Nightly** (локально, без аккаунта): `powershell -ExecutionPolicy Bypass -File tools\build-xpi.ps1` → `dist\gptgraber-<ver>.xpi`; в `about:config` выставить `xpinstall.signatures.required=false`; поставить через `about:addons → Install Add-on From File`.
- **Любой Firefox** (подпись AMO unlisted): `npm install`, задать `WEB_EXT_API_KEY`/`WEB_EXT_API_SECRET` (ключи с addons.mozilla.org), `npm run sign:ext`. Артефакты — в `dist/` (в гит не коммитим, см. `.gitignore`).

**Быстрое обновление постоянного расширения (dev-цикл):** `powershell -ExecutionPolicy Bypass -File tools\update-ext.ps1` — авто-версия (база из манифеста + секунды от 2024-01-01, исходный `manifest.json` НЕ трогается) + сборка `dist\gptgraber-dev.xpi` + открытие его в Firefox на установку (предпочитает уже запущенный экземпляр). Остаётся клик «Add»; таб ChatGPT перезагрузит сам `background.js` по `runtime.onInstalled`. Стабильный ID `gptgraber@local` → новый `.xpi` встаёт поверх старого. Опции: `-NoLaunch`, `-FirefoxPath`. В отличие от `build-xpi.ps1` (релизный артефакт с реальной версией) — это инструмент частых правок.

Проверка синтаксиса JS: `node --check content.js` / `background.js`.

## 5. WS-протокол (кратко; полностью — docs/PROTOCOL.md)

`ws://<host>:17890/`, сообщения `{ "type": "...", "payload": "..." }`, UTF-8. Роль соединения сервер определяет по `hello`: payload со словом `controller` → сетевой клиент, иначе → Firefox.

- Firefox → сервер: `hello`, `text` (полный innerText композера), `recording` (запись реально началась), `ready` (таб ChatGPT готов к диктовке — ответ на `ensureTab`).
- сервер → Firefox: `hello`, `ack`, `ensureTab` (открыть/проверить таб ChatGPT, ответить `ready`), `mic` (клик микрофона), `stop` (клик птички), `clear` (очистить композер). Адресно — `SendToFirefox`. Готовность таба внутри расширения — внутренний пинг `checkReady` к `content.js` (НЕ часть WS-контракта).
- сетевой клиент → сервер: `hello` (`"controller"`), `start`, `stop`.
- сервер → сетевой клиент: `hello`, `inject` (текст диктовки владельцу сессии). Адресно — `SendToController` по Id соединения.

## 6. Ключевые технические решения и ГРАБЛИ (не наступать повторно)

1. **ChatGPT пересоздаёт DOM-узлы.** `#prompt-textarea` (ProseMirror) и все кнопки композера пересоздаются после каждого манёвра; **во время записи `#prompt-textarea` вообще исчезает**. СТАБИЛЬНА только `<form>` (класс содержит `composer`).
   → Поэтому: НЕ кэшировать ссылки на узлы/кнопки. Композер читать поллингом (`setInterval`, каждые 250 мс), кнопки искать заново от формы при каждом клике. MutationObserver на конкретном узле — НЕ годится (узел умирает).

2. **Селекторы ChatGPT DOM** (aria-label на английском даже при русском UI):
   - форма: `form`, у которой `className` содержит `composer`;
   - микрофон: `button[aria-label="Start dictation"]`;
   - птичка: `button[aria-label="Submit dictation"]`;
   - отмена: `button[aria-label="Cancel dictation"]`;
   - отправка промпта боту — `#composer-submit-button`, фолбэки `button[data-testid="send-button"]` → `button[aria-label="Send prompt"]` (id/testid могут сменить, label может быть локализован): её НЕ кликаем, но используем как ИНДИКАТОР исхода распознавания (есть после диктовки ⇔ распознался текст; тишина ⇒ её нет, вместо неё кнопка голосового режима). См. §6.19(г).
   Есть фолбэк-поиск по подстроке `dictation|диктов`. Индикатор «запись идёт» = наличие кнопки `Submit`/`Cancel dictation`.

3. **Вставка в Electron/Chromium — только синтез ввода.** `WM_PASTE`/`WM_CHAR`/`WM_SETTEXT` не доходят до Chromium-виджета. Используем буфер обмена + `Ctrl+V` через `SendInput` **скан-кодами** (`KEYEVENTF_SCANCODE`). Посимвольный ввод не годится (кириллица/Unicode, скорость).

4. **Защита от кражи фокуса в Windows.** `SetForegroundWindow` работает, только если процесс уже на переднем плане. Обход в `Injector.ForceForeground`: `SPI_SETFOREGROUNDLOCKTIMEOUT=0` → `AttachThreadInput` (склейка очередей ввода с foreground- и target-потоком) → `SetForegroundWindow`/`SetFocus` → обязательная отвязка и возврат таймаута. Используется и для подъёма Firefox на старте, и перед Ctrl+V (гарантия, что цель на переднем плане).

5. **Низкоуровневый хук должен быть быстрым.** В колбэке `KeyboardHook.HookProc` НЕЛЬЗЯ залипать (таймаут LL-хука) — только `PostThreadMessage` себе в очередь; тяжёлую работу делает цикл сообщений. Делегат хука хранится в поле (иначе GC съест). Хук ставится на главном потоке (у него цикл сообщений).

6. **Тяжёлая работа — на потоке цикла сообщений.** И хоткей-хук, и приходящие по WS события (`text`, `recording`, а также `start`/`stop` от сетевых клиентов) маршалятся в главный поток (`WM_APP_TOGGLE`, `WM_APP_INJECT`, `WM_APP_FOCUS_BACK`, `WM_APP_CTRL_START/STOP` через `PostThreadMessage`). У этого потока есть очередь ввода — важно для `AttachThreadInput`. Сам WS-сервер живёт в фоне. Id сетевого клиента (для маршрутизации текста обратно) передаётся через `wParam` сообщения — не управляемый объект, а int, поэтому проходит через `PostThreadMessage` без проблем; машина состояний (`_state`/`_ownerId`) трогается только этим потоком.

7. **Гейт авто-вставки.** Сервер вставляет текст только если `_pendingInject` (выставляется на СТОПе). Чужие/случайные изменения композера не вставляются.

8. **Кодировка консоли.** В `Program.Main` стоит `Console.OutputEncoding = UTF8`, иначе кириллица в консоли Windows = `?`.

9. **WS из расширения, не Tampermonkey.** WS до localhost открывается из background-контекста расширения, который НЕ подчиняется CSP страницы ChatGPT. Background — persistent MV2 (MV3 event-page усыпил бы WS).

10. **Команды серверу в табы** идут через `browser.tabs.query({}) + tabs.sendMessage` (без url-фильтра → доп. прав не нужно; не-наши табы молча отваливаются).

11. **Firefox не стартует захват микрофона для окна БЕЗ фокуса.** Программный `.click()` по «Start dictation» проходит, UI открывается, но реальная запись висит, пока окно FF в фоне (защита от скрытой записи). Раз начавшись при фокусе, запись продолжается и в фоне. → Обход на старте: сервер находит окно FF (`WindowFinder`: `MozillaWindowClass` + заголовок содержит «ChatGPT»), на мгновение выносит его вперёд (`Injector.ForceForeground`), шлёт `mic`. Расширение, увидев индикатор записи, шлёт `recording` → сервер возвращает фокус в окно, активное до старта (`_returnWindow`). Страховка: `Timer` на `Config.RecordingFocusTimeoutMs` (если `recording` не пришёл — вернуть фокус всё равно). Возврат фокуса маршалится через `WM_APP_FOCUS_BACK`.

12. **Хоткей-клавиша — по скан-коду, не по vkCode.** `vkCode` зависит от раскладки: одна и та же физическая клавиша на немецкой раскладке = `VK_Y` (0x59), на русской = `VK_Z` (0x5A). Скан-код аппаратной позиции одинаков всегда. Поэтому клавиша «+буфер» ловится по `scanCode == 0x2C` (`Native.SCAN_KEEPBUFFER`), а модификаторы Ctrl/Win — по vk (они от раскладки не зависят).

13. **Хаб: роли, владелец сессии, единственный ресурс.** Сервер различает соединения по `hello` (`controller` → сетевой клиент, иначе Firefox) и шлёт адресно: `mic/stop/clear` — только Firefox-соединениям (`SendToFirefox`), `inject` — конкретному клиенту (`SendToController` по `Id`). НЕ возвращать `Broadcast` (рассылал бы команды диктовки и контроллерам). Диктовка — один общий Firefox: `_ownerId` фиксируется на СТАРТЕ, текст уходит этому владельцу; пока `Recording`, чужие старты игнорируются. `_pendingInject` сбрасывается на старте (страховка от «зависшего» прошлого СТОПа). Окно-цель и буфер для удалённого владельца — забота клиента (он захватывает своё окно по `stop` и вставляет у себя); сервер удалённый HWND не знает. Доступ по сети к `http://+:17890/` требует разового `netsh http add urlacl` (или запуска от админа) и правила брандмауэра — иначе `HttpListener` бросит «отказ доступа».

14. **Smart App Control (SAC) блокирует неподписанный apphost.** На Windows 11 с включённым SAC свежесобранный `VoiceBridge.exe` (apphost) не запускается («Application control policy has blocked this file») — и `dotnet run` тоже, т.к. он запускает этот exe. SAC доверяет только подписанным/репутационным бинарям, отдельных исключений не даёт, а выключается необратимо (до переустановки Windows). Обход без отключения SAC: `UseAppHost=false` в csproj — exe не генерируется, запуск идёт через доверенный `dotnet.exe` (`dotnet run` / `dotnet <dll>`). Не возвращать apphost, пока проект не подписывается. На клиентских машинах с SAC — так же: нужен .NET-рантайм и запуск DLL через `dotnet`.

15. **Firefox не стартует захват микрофона в ФОНОВОМ (невыбранном) табе** — это отдельная от §6.11 защита: §6.11 про фокус *окна*, а эта про *активный таб* внутри окна. Если в Firefox выбран другой таб (не ChatGPT), программный `.click()` по «Start dictation» проходит, UI открывается, но запись висит. Сервер таб не контролирует — это умеет только расширение (`browser.tabs`). → Обход целиком в `background.js`: на команду `mic` сперва `tabs.update(id, {active:true})` + `windows.update(winId, {focused:true})` (await), и ТОЛЬКО ПОТОМ рассылка `mic` на клик. Id таба ChatGPT background берёт из `sender.tab.id` сообщений content script'а (тот шлёт `register` при загрузке, плюс id обновляется на каждом `text`/`recording`) — поэтому url-фильтрованный `tabs.query` (которого §6.10 избегает) и доп. разрешения НЕ нужны: `tabs.update`/`windows.update` без смены `url` прав не требуют. Закрытие таба ловится `tabs.onRemoved` (сбрасываем id; новый таб зарегистрируется сам). `register` — ВНУТРЕННЕЕ сообщение content→background (`runtime.sendMessage`), НЕ часть WS-протокола, в `docs/PROTOCOL.md` его нет. `stop`/`clear` активации не требуют (submit идёт в фоне, clear намеренно фоново-безопасен через `execCommand`). **Возврат прежнего таба:** `background.js` запоминает таб, который был активен (`prevActiveTabId`), и возвращает его ПОСЛЕ старта захвата — по сигналу `recording` плюс ЗАДЕРЖКА `RESTORE_PREV_TAB_DELAY_MS` (2.5 с). Грабля: `recording` приходит по появлению UI диктовки, РАНЬШЕ реального старта захвата (~1.5 с; сервер ровно поэтому держит фокус окна `MinFocusHoldMs`) — мгновенный возврат таба гарантированно срывал захват: UI висит, «запись идёт», а записи нет, стоп даёт «тишину» (проявлялось, когда таб ChatGPT был восстановлен сессией, а активен другой таб). Возврат — ВСЕГДА, в т.ч. когда таб ChatGPT создали мы сами (решение пользователя 2026-06-11; прежнее «остаёмся в созданном табе» отменено). Так как при старте таб ChatGPT может стать активным ещё на этапе ensureTab (создание active-табом, пробуждение ленивого), prev запоминается ДО этого (`rememberPrevActiveTab`, уже запомненный не перезаписывается); сброс «хвостов» прошлой сессии — в начале ensureTab(start) и в `activateChatGptTab`. Раз начавшись при активном табе, запись идёт и в фоновом. Возвращаем именно ТАБ; фокусом ОС-окна по-прежнему рулит сервер (§6.11) — это независимые вещи. Если `recording` не пришёл — таб не возвращаем (вероятно, захват не стартовал, пусть ChatGPT остаётся видимым для диагностики).

16. **Авто-подготовка Firefox/таба ChatGPT (Idle→Preparing→Recording).** Старт диктовки больше НЕ бьёт `mic` сразу: сервер сначала гарантирует, что Firefox с готовым табом ChatGPT существует, и лишь по сигналу `ready` поднимает окно и шлёт `mic`. Кто что делает (по тому, что каждый может наблюдать): **«нет таба, но FF жив»** решает РАСШИРЕНИЕ (только оно знает о табах) — на `ensureTab` ищет таб (`tabs.query` по url, для чего добавлены host-permissions `*://chatgpt.com/*`, `*://chat.openai.com/*`), при отсутствии открывает ПАССИВНО (`active:false`, чтобы не дёргать пользователя на прогреве) и шлёт `ready` ПО ГОТОВНОСТИ. Готовность зависит от того, наш таб или нет (грабля, см. ниже): **свежесозданный** таб = дождаться полной загрузки страницы (`tabs.get(id).status==="complete"`, хелпер `waitTabLoaded`) → кнопка «Start dictation» (`waitTabReady`/`checkReady`) → осадка `CREATED_TAB_SETTLE_MS` (~1 c) и только потом `ready`; **уже существующий** таб уже прогрет — ждём только кнопку. Грабля: у фонового свежесозданного таба кнопка «Start dictation» может появиться РАНЬШЕ, чем страница реально готова к захвату — поэтому одного `checkReady` мало, нужны загрузка + осадка. Слепой задержки перед кликом в обработчике `mic` НЕТ — вся подготовка свежего таба сделана здесь, до `ready`. **«FF не запущен»** решает СЕРВЕР (только он жив, когда браузера нет) — `FirefoxLauncher` находит `firefox.exe` (реестр `App Paths`→`StartMenuInternet`→Program Files) и запускает его БЕЗ URL (см. §6.19д — URL давал дубль при восстановлении сессии); расширение подключится, подготовит таб и пришлёт `ready`. `RequestStart`: FF на связи → `ensureTab`+таймаут `PrepareReadyTimeoutMs`; FF запущен, но расширение молчит → ждём подключения (таймаут `PrepareLaunchTimeoutMs`); FF не запущен → `Launch`. По `ready` (`HandleReady`) — `BeginRecording` (если ждали старта); прогревный `ready` (старт сервера/реконнект) игнорируется. Таймаут `HandlePrepareTimeout` → `Idle` + понятное сообщение. **`ensureTab` шлётся и при подключении расширения** (`OnFirefoxConnected`), и **на старте сервера** (`PrepareFirefoxAtStartup` поднимает FF, если не запущен) — это прогрев БЕЗ записи. `_returnWindow` запоминаем в `RequestStart` (пока пользователь ещё в рабочем окне), а не в `BeginRecording`. Окно-подъём и страховочный таймер `_focusBackTimer` (§6.11, `RecordingFocusTimeoutMs`) стартуют только в `BeginRecording`, т.е. ПОСЛЕ готовности таба — иначе создание таба (5–10 с) превысило бы таймаут возврата фокуса и фокус вернулся бы ещё до старта записи. Отключение авто-запуска: `--no-autolaunch`. **Грабля-предупреждение:** ВРЕМЕННОЕ расширение (`about:debugging`) слетает при перезапуске браузера — значит, ветка «запустить полностью закрытый Firefox» даст браузер БЕЗ расширения (мост не подключится, сработает таймаут с сообщением). Эта ветка рассчитана на ПОСТОЯННО установленное расширение; кейс «FF жив, таба нет» работает при любой установке.

17. **Системный индикатор статуса (StatusOverlay) — окно, которое НЕ должно красть фокус.** Маленькая «пилюля» поверх всех окон показывает фазу диктовки: оранжевая звёздочка-мерцалка «Готовлю ChatGPT…» (Preparing) → оранжевый пульсирующий микрофон «Включаю микрофон…» (ready пришёл, mic отправлен) → ЗЕЛЁНЫЙ пульсирующий микрофон «Идёт запись» (по РЕАЛЬНОМУ сигналу `recording`: wParam=1 в `WM_APP_FOCUS_BACK` отличает его от страховочных таймеров) → синяя звёздочка «Распознаю…» (стоп) → галочка «Вставлено» / крестик-ошибка (гаснут сами по `WM_TIMER`). Ключевые решения: (а) стили `WS_EX_NOACTIVATE | WS_EX_TOPMOST | WS_EX_TOOLWINDOW | WS_EX_LAYERED | WS_EX_TRANSPARENT` + показ только `SetWindowPos(SWP_NOACTIVATE)` — окно не активируется и пропускает клики насквозь; (б) живёт на ГЛАВНОМ потоке — его цикл сообщений уже зовёт `DispatchMessage`, отдельный UI-поток не нужен; все методы дёргать только с этого потока; (в) делегат WndProc — в статическом поле (иначе GC съест, как у хука §6.5); (г) значки — глифы встроенного шрифта «Segoe MDL2 Assets» (PUA-символы прямо в исходнике StatusOverlay.cs — НЕ «пустые строки», не трогать!); символ ожидания — «звёздочка» из кадров-дингбатов ·✢✳✶✻✽ (НЕ GDI-дуга `Arc` — она без сглаживания, выходила рваной), анимация `WM_TIMER` каждые 80 мс; рисуются глифы НЕ DrawString'ом, а контуром через `GraphicsPath.AddString` + центрирование по фактическому ink-прямоугольнику (`GetBounds`) — DrawString центрирует строчную коробку шрифта, в которой значки сидят несимметрично, и каждый глиф требовал ручного сдвига; масштаб per-глиф (`CheckScale`/`CrossScale`) — галочка/крестик занимают весь em-квадрат и без поджатия выглядят крупнее микрофона; шрифт ПОДПИСИ — первый доступный из `TextFontChain` (Google Sans → Roboto → Segoe UI): сперва приватная папка `fonts/` рядом с DLL (`PrivateFontCollection.AddFontFile` — работает без установки шрифта в систему; у семьи без Bold берётся Regular, иначе конструктор Font бросает), затем установленные; (д) якорь — в `CaretGapX` (3 px) ПРАВЕЕ каретки активного окна, центр по её высоте (`GetGUIThreadInfo.rcCaret`+`ClientToScreen`); Electron/Chromium каретку НЕ отдают (`hwndCaret=0`) → фолбэк: НИЗ-ЦЕНТР рабочей области монитора с окном-целью, «как у Whisper» (решение пользователя; «верх окна» ему не понравился); позиция кламптся в рабочую область (`MonitorFromPoint`); (е) индикатор только для ЛОКАЛЬНОГО владельца — сетевым клиентам не транслируется (v1; потребовало бы строку в протоколе); (ж) перезаякоривание на СТОПе (`Show` с окном-целью), фазы без движения — `Set`; (з) «Распознаю…» имеет страховочный авто-скрыв 30 с — если текст не пришёл, пилюля не висит вечно. Флаги: `--overlay-compact` (только значок-кружок), `--no-overlay`, `--overlay-colors key=RRGGBB,…` (все цвета в `Config.OverlayCol*`: bg/border/text/wait/rec/busy/err; дефолт — тёмная пилюля с белой рамкой). **Отрисовка — PER-PIXEL ALPHA (`UpdateLayeredWindow`), НЕ регион+WM_PAINT:** пилюля каждый кадр рендерится в ARGB-битмап через GDI+ (`System.Drawing.Common`, единственный NuGet-пакет проекта; `SmoothingMode.AntiAlias`) и целиком отдаётся окну — гладкие полукруглые торцы и рамка (`GraphicsPath` из двух дуг). Грабля, из-за которой так: `SetWindowRgn`+GDI-рисование НЕ сглаживаются — рваные края рамки и формы, пользователь забраковал; GDI-дугу спиннера забраковал ещё раньше (та же причина). После `UpdateLayeredWindow` нельзя мешать `SetLayeredWindowAttributes` — режимы взаимоисключающие. Битмап пересоздаётся под ширину текущего текста (пилюля «дышит»), позицию задаёт сам `UpdateLayeredWindow`, `SetWindowPos` — только topmost+показ.

18. **Локализация статичных надписей (Lang.cs + lang/*.json).** Все надписи консоли и индикатора — через `Lang.T("ключ", args…)`; шаблоны string.Format (`{0}`, `{1:X}`). Модули: `lang/en.json`, `lang/de.json`, `lang/ru.json` (плоские словари; копируются в выход через `None Update` в csproj, читаются из `AppContext.BaseDirectory/lang`). Выбор языка: язык UI Windows → `--lang en|de|ru` перебивает. **Грабля:** `CultureInfo` для детекта НЕ работает — в csproj `InvariantGlobalization=true` (все культуры инвариантные); язык берём через P/Invoke `GetUserDefaultUILanguage()` (LANGID, младшие 10 бит: 0x07 de, 0x19 ru, иначе en). Фолбэк при отсутствии ключа: выбранный язык → en → сам ключ (не падаем). `--lang` применяется прямо в ParseArgs (последующие ошибки разбора уже на нём); `Lang.Init()` зовётся в начале `Main` ДО разбора аргументов. Хвосты-суффиксы условных фраз — отдельные ключи (`suffix.keep_buffer`, `inject.kept_suffix`).

19. **Сбои старта/записи: launch-grace, активный таб, abort-путь (исправления по итогам теста).** Три связанные грабли. (а) **Двойной запуск Firefox**: прогрев на старте сервера (`PrepareFirefoxAtStartup`) и старт записи (`RequestStart`) — независимые вызовы `FirefoxLauncher.Launch`; в гонке они открывали ДВА окна браузера. Лечение: launch-grace — `FirefoxLauncher` помнит момент своего запуска FF (`IsStartingUp`, `Config.FirefoxLaunchGraceMs` = 30 с), повторный `Launch` в этот период подавляется, а `RequestStart` при `IsStartingUp` идёт в ветку «жду подключения расширения», не запуская ничего. (б) **Пассивный таб открывается очень долго**: фоновые табы Firefox грузит с пониженным приоритетом, плюс расширение перед созданием таба 3 с ждало «не появится ли чужой». Лечение: `ensureTab` получил payload-подсказки (`start`/`warmup` + `,cold`, см. PROTOCOL.md): под запись (`start`) новый таб создаётся АКТИВНЫМ, ожидание чужого таба разделено на hot (0.8 с — обычный случай) и cold (12 с — FF только что поднят сервером, его стартовый таб ещё не виден в `tabs.query`, не плодить дубль). «Свежесть» страницы (нужны полная загрузка + осадка) определяется не только «мы создали таб», но и `tab.status !== "complete"` — покрывает стартовый таб свежеподнятого FF. (в) **Зависание на «Распознаю…», когда захват не стартовал** (нет звука/микрофона): СТОП слал Submit и взводил ожидание текста, которое не приходило никогда. Лечение: сервер помнит `_recordingConfirmed` (сигнал `recording` этой сессии); СТОП без подтверждения идёт по abort-пути `AbortRecording` — шлёт **`cancel`** (клик «Cancel dictation»), не вооружается на текст, сразу `Idle`, индикатор «Запись не стартовала». Плюс общая страховка `_injectTimer` (`Config.InjectTimeoutMs` = 20 с): текст после СТОПа не пришёл — разоружаемся (`HandleInjectTimeout`, `WM_APP_INJECT_TIMEOUT`), поздний `text` не вставится. (д) **Холодный старт при ВОССТАНОВЛЕНИИ СЕССИИ Firefox**: запуск `firefox.exe <url>` давал дубль (таб из сессии + таб из командной строки), а восстановленный сессией таб — «ленивый» (`discarded`): `tabs.query` его видит, но страница не загружена, content.js в нём не живёт, `checkReady` молчит → ничего не стартовало. Лечение: сервер запускает FF **без URL** (`FirefoxLauncher.Launch()`; таб ChatGPT — целиком забота `ensureTab`, одна точка истины), а расширение будит ленивый таб (`t.discarded` → для старта `tabs.update(active:true)`, для прогрева `tabs.reload`) и ведёт его как «свежую страницу» (загрузка + кнопка + осадка). (г) **Запись шла, но в ней ТИШИНА** (микрофон отвёрнут и т.п.): `recording` был, СТОП легитимно шлёт Submit, но ChatGPT распознаёт пустоту — композер остаётся пустым, `text` не придёт, «Распознаю…» висело до таймаута. Лечение: `content.js` после Submit следит за исходом (`watchTranscriptionOutcome`) и ждёт ОДНОЗНАЧНОГО сигнала (подсказки пользователя): появилась кнопка отправки `#composer-submit-button` (или текст) ⇔ распознался текст — выход молча (текст отправит поллер); вернулась кнопка ГОЛОСОВОГО РЕЖИМА (`composer-speech-button` / aria-label со словом voice; при тишине она встаёт на место кнопки отправки) ⇔ «пусто» → серверу **`empty`**; сервер (`HandleEmpty`, `WM_APP_EMPTY`) немедленно разоружается, индикатор «Пусто — текста нет». **Грабля:** «пусто» НЕЛЬЗЯ определять тупым таймером «композер пуст N секунд» — медленное распознавание давало ЛОЖНОЕ «пусто», текст приезжал позже, оставался в композере и примешивался к следующей диктовке (ChatGPT дописывает, не заменяет; склейка при этом сознательно НЕ чистится — пользователь решил, что «не теряется» важнее). Таймер (8 с без любых сигналов) — только фолбэк на случай смены селекторов.

20. **Ctrl+Win как универсальная ОТМЕНА в промежуточных фазах (локальный контроллер).** Пара «старт/стоп» — только из Idle (старт) и из подтверждённой записи (стоп). Во ВСЕХ остальных промежутках повторный Ctrl+Win = отмена всего и возврат в Idle (`HandleToggle`, §6.20): (а) `Preparing` (готовим FF/таб, владелец локальный) → `CancelLocalPrepare` — отмена старта; уже отправленный `ensureTab` не вредит (таб подготовится «прогревом», `ready` проигнорируется); если подготовка шла под СЕТЕВОГО владельца — локальное нажатие по-прежнему «занято»; (б) `Recording` без подтверждённого захвата → `AbortRecording` (§6.19в); (в) `Idle`, но `_pendingInject` (распознавание после СТОПа ещё идёт) → `CancelPendingInject` — вставка разоружается, индикатор «Отменено»; поздний `text` отменённого распознавания НЕ вставляется, но композер по его приходу вычищается (`_discardNextText` → `clear`), иначе он примешался бы к следующей диктовке; «empty» после отмены просто сбрасывает флаг. `_discardNextText` сбрасывается на старте новой записи. Для сетевых клиентов аналогичной отмены пока нет (их СТОП во время подготовки уже отменяет её — `HandleCtrlStop`).

- Контракт WS меняем ТОЛЬКО синхронно: `docs/PROTOCOL.md` + `WsServer.OnMessageAsync` + `WsClient.OnMessage` + обработчики в `content.js`/`background.js`.
- Проверять после изменений: `dotnet build` для C#, `node --check` для JS.
- В гит не коммитим `bin/`, `obj/`, `.vs/` (см. `.gitignore`).

## 8. Автономная версия на WebView2 («GPT Grabber»)

Отдельное самодостаточное приложение в `poc-webview2/` (проект `WebView2Poc`, продукт «GPT Grabber»). Встраивает ChatGPT через **WebView2** (Chromium) и управляет страницей **в процессе** (инъекция JS), без внешнего Firefox, расширения и сетевого хаба. Переиспользует идеи Firefox-версии: хук хоткея, вставку (Injector/Clipboard), индикатор-пилюлю (StatusOverlay), грабли §6.2/§6.3/§6.4. WS-протокол НЕ используется (PROTOCOL.md → раздел «WebView2-версия»).

```
poc-webview2/
├─ Program.cs    entry: скрытие консоли (перезапуск без окна), single-instance, парс --lang/--tray/--no-beep
├─ MainForm.cs   окно (WebView2 сверху + лог снизу), трей, машина Idle→Starting→Recording→Stopping, JS-скрипты, справка «?»
├─ Overlay.cs    индикатор-«пилюля» (перенос StatusOverlay; процесс DPI-unaware)
├─ Hotkey.cs     низкоуровневый хук Ctrl+Win / +Y(скан 0x2C) / +Alt; постит в HWND формы (WM_APP_TOGGLE/REPASTE)
├─ Interop.cs    P/Invoke: хук, SendInput(Ctrl+V), буфер, FeedbackBeep(Beep), single-instance (RegisterWindowMessage/HWND_BROADCAST)
├─ Lang.cs       en/de/ru, словари в КОДЕ (не JSON); --lang перекрывает систему
├─ Diag.cs       данные рядом с программой: AppContext.BaseDirectory/data (профиль WebView2 + poc.log)
├─ icons/        ico2.png — источник иконки (make-icon.ps1); прочее — наброски (в гит попали)
└─ fonts/        приватные .ttf для подписи пилюли (как у сервера; в гит не коммитим)
```

### Ключевые решения (доп. к §6)

1. **SAC и окно консоли.** Свой apphost не собираем (`UseAppHost=false`, §6.14) → запуск через `dotnet.exe` (консольный хост). На Win11 с Windows Terminal `ShowWindow(GetConsoleWindow())` не прячет окно (GetConsoleWindow отдаёт скрытую псевдоконсоль, видимое окно держит Terminal). Решение: приложение на старте **перезапускает себя без окна** (`Process.Start` + `CreateNoWindow`), пометив дочерний процесс env `GPTGRABBER_NOCONSOLE` (чтобы не зациклить), и выходит. Короткое мелькание при старте неустранимо без подписанного exe.
2. **Один экземпляр.** Именованный мьютекс `Local\GptGrabber.SingleInstance`; держит его только реальный экземпляр (после relaunch-для-консоли создаёт дочерний). Второй запуск шлёт первому broadcast `RegisterWindowMessage("GptGrabber.ShowExistingInstance.v1")` → `MainForm.WndProc` → разворот из трея (доходит и до скрытого окна). Гонку решает `createdNew`.
3. **Данные рядом с программой** (`Diag.Dir = AppContext.BaseDirectory + "data"`): профиль ChatGPT (WebView2) и лог — в `{папка}\data`. Установленная версия → `%LOCALAPPDATA%\GptGrabber\data` (per-user, писемо); Debug → `bin\…\data`. Деинсталлятор сносит. Для per-machine пришлось бы вернуть в %LOCALAPPDATA% (Program Files не писемо).
4. **Управление страницей — инъекция JS** (`ExecuteScriptAsync`), не WS. Скрипты-операции — в PROTOCOL.md (раздел WebView2). Селекторы/грабли DOM те же (§6.1/§6.2/§6.19). §6.11/§6.15 (фокус окна/таба для микрофона) НЕ нужны: захват идёт и в свёрнутом окне (WebView не «спит» благодаря `--disable-background-timer-throttling …`), активного «таба» нет.
5. **Скрытие лишних блоков ChatGPT** (`HideExtrasScript`): CSS-правило в `<head>` (живёт после перерисовок React — узлы НЕ удаляем) прячет соседей `#thread-bottom`: блок-после (`#thread-bottom + div`) и заголовок-до (`*:has(+ #thread-bottom-container)`).
6. **Звук вставки** (`Win32.FeedbackBeep`): свой тон через kernel32 `Beep` на фоновом потоке (системный `MessageBeep` звучал «как ошибка»). `--no-beep` глушит.
7. **Прочее:** окно компактное 740×550; верхний отступ текста в логе/справке — стартовой пустой строкой (скролл и кнопку «?» не трогает); справка по «?» — прокручиваемое поле, полоса лога при открытии справки временно растёт.

### Установщик (Inno Setup)

`installer/GptGrabber.iss` + сборка одной командой `tools/build-installer.ps1`: publish framework-dependent → вложенный **портативный .NET 10 runtime** (NETCore.App **и** WindowsDesktop.App — нужен из-за WinForms; берём из локального `C:\Program Files\dotnet`) → бутстраппер **Edge WebView2 Runtime** (кэш `tools/cache/`, ставится только если рантайма нет) → ISCC. Свойства: per-user в `%LOCALAPPDATA%\GptGrabber`, без админа, БЕЗ шагов мастера (`DisableWelcomePage/DirPage/ReadyPage`, `ShowLanguageDialog=no`); запуск через вложенный подписанный `dotnet.exe` + DLL (SAC); `[UninstallDelete]` сносит `{app}\data`. Иконка — `installer/app.ico` (генерится `make-icon.ps1` из `icons/ico2.png`). Версия — из `package.json`. Выход — `dist/GptGrabber-Setup-<ver>.exe`.

**Грабли установщика:** (а) `.ps1` и `.iss` с кириллицей сохранять в UTF-8 **с BOM** (PowerShell 5.1 и Inno иначе читают как ANSI → кракозябры/парс-ошибки); (б) в Pascal-комментариях `.iss` нельзя `{`/`}` (обрывают комментарий); (в) `Get-Content -Raw` в PS5.1 читает UTF-8 как ANSI — файлы (манифест и т.п.) читаем через `[System.IO.File]::ReadAllText`; (г) большой `setup.exe` (~52 МБ, с рантаймом) коммитим в git осознанно — растит историю.
